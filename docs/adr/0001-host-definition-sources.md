# Host definition sources

The language server provides Office object model intelligence from `HostDefinition`s supplied by configured `HostApplication`s. Each supported host application ships with bundled catalog data for immediate and portable behavior, centered on high-frequency root object types and enriched with available `CallableSignature` metadata. It also uses a per-host user cache under `~/.vba-language-server/host-catalogs/` when available.

Host catalog refresh, COM object member discovery, and `HostTypeLibraryDiscovery` are separate concerns. COM object member discovery uses Office application automation and representative objects on Windows when a `HostApplication` is installed and selected; this refresh is suited to discovering available root and member names for the user's installed Office version. `HostTypeLibraryDiscovery` reads Office type library metadata for host enums, enum members, standalone constants, callable signatures, parameter metadata, return types, values, and documentation where available. Either discovery source may enrich the per-host cache independently; type library-derived metadata can also be checked into the bundled catalog snapshot so non-Windows or non-Office environments still have useful fallback metadata. Enum discovery is limited to enum type definitions, and standalone constant discovery is limited to type-library module constants so object member discovery does not blur into constant discovery.

Discovered definitions take precedence because they match the user's installed Office version, while bundled definitions remain the fallback so startup and tests do not depend on Office or COM availability. COM and type-library discovery failures do not produce user-facing warning popups when fallback catalog data is available. External reference libraries such as DAO and ADO are separate from host application object models and are not enabled by Access host support.
