# Workbook-backed project command model

`vba-dev` will model Office macro automation as a Windows-only, workbook-backed command-line workflow centered on a `vba-project.json` project manifest. The manifest identifies the project and primary document, supplies command defaults, and anchors path resolution so commands can behave consistently from project subdirectories and future multi-document projects.

## Decision

Project manifests use `schemaVersion: 1`, are generated as UTF-16LE with BOM, and are the source of truth for resolving the project root, primary document, document source set, bin output, publish output, and default command options. The reader accepts UTF-16LE with BOM and UTF-8 inputs for compatibility, but generated or rewritten manifests normalize to UTF-16LE with BOM.

Project creation follows the `dotnet new <TEMPLATE>` shape. The initial template is `excel`, so users create a project with `new excel`; future Office document kinds can become additional template names. `--output` selects the project root directory. `--name` selects the generated project and document base name, and when omitted it is derived from the output directory name or current directory name.

The initial source layout is document-oriented even though only one primary Excel document is supported at first: `src/<document-name>/` contains the source template `.xlsm` and exported VBA source. Generated workbook outputs do not need a document-name subdirectory when the workbook file name already carries the document name, so new projects default to `bin/<document-name>.xlsm` and `publish/<document-name>.xlsm`. Existing manifests that name paths such as `bin/<document-name>/<document-name>.xlsm` and `publish/<document-name>/<document-name>.xlsm` remain valid because `vba-project.json` is the source of truth for output path resolution. Document-level commands accept `--project` and `--document`; omitted values resolve by walking upward to the nearest `vba-project.json` and then using the manifest's `primaryDocument`.

When `new excel` can copy required initial CommonModules from the canonical repository, it records those entries under the generated document's `commonModules` manifest list as well as copying their source files into the generated **CommonModulesDirectory** using each entry's extension-including file name. Required initial baseline entries are recorded with `requested: true`; dependencies pulled in by that baseline are recorded with `requested: false`. Initial projects therefore start with a manifest-backed installed CommonModules set rather than loose copied files.

Common option aliases follow .NET CLI conventions where they do not create ambiguous command surfaces. `--name` has `-n`, `--output` has `-o`, `--document` has `-d`, and `--format` has `-f`. `--project` has no short alias because it is a context selector rather than an output or resource name. `--from` and `--to` also have no short aliases because `-f` belongs to `--format` and `--to` is intentionally distinct from `--output`.

Common modules are copied from a canonical `common_modules_repo`. `new` creates a usable project even when the repository is missing, but reports a warning; `doctor` reports the missing repository as an environment or project problem. `doctor` also checks manifest-defined **VbaProjectReferences** for every document without changing workbooks. For **DocumentSourceSet** source files, `doctor` reports case-insensitive duplicate extension-including `.bas`, `.cls`, or `.frm` file names as failures because flat source identity cannot map them to a single workbook import target. `doctor` reports a warning for any `.frx` file that has no matching same-directory `.frm` but has a same-name `.frm` elsewhere in the source set, because that sidecar looks displaced rather than intentionally standalone. `.frx` files that have no same-name `.frm` anywhere in the source set are ignored. For **InstalledCommonModule** entries, `doctor` verifies that names exist in the CommonModules manifest, that dependencies required by `requested: true` roots are present, and that copied source files are consistent with the manifest. Missing or unknown required entries are failures. Unreachable `requested: false` entries and source file drift start as warnings. CommonModules dependencies and classifications are expected to come from a machine-readable manifest owned by `xls-common-devtools`, not from hard-coded `vba-dev` rules or scraped product documentation.

Resource-specific operations follow .NET CLI-style noun-first command groups where possible instead of using one top-level verb that infers a resource from positional names. CommonModules are added with `common-module add`, listed with `common-module list`, and updated with `common-module update`. `common-module add` accepts one or more CommonModules names in a single invocation and is scoped to one document source set. `common-module list` is also document-scoped, defaults to the primary document when `--document` is omitted, identifies the selected document in its output, defaults to human-readable text, and supports `--format json` for automation. `common-module update` updates already-installed CommonModules entries as a project-wide operation; the initial implementation does not need per-module update selection. `common-module remove` is not part of the initial command surface because dependency ownership, shared dependencies, and locally edited copied files need separate deletion semantics. **VbaProjectReferences** use `reference add`, `reference list`, and `reference remove`. These reference commands are scoped to one document definition. `reference add` and `reference remove` accept one or more reference names in a single invocation. `reference list` identifies the selected document in its output, defaults to human-readable text, and supports `--format json` for automation. Because the command surface is not released yet, earlier top-level forms such as `add <module>`, `add module`, `add reference`, `update`, and `remove reference` do not need compatibility support. `reference remove` removes **VbaProjectReferences** from the manifest only and succeeds for references that are already absent.

**InstalledCommonModule** entries belong to the document definition in `vba-project.json`, under `documents.<document-name>.commonModules`, as objects with a `name` field. CommonModules names are extensionless module base names; the CommonModules manifest resolves those names to actual `.bas`, `.cls`, `.frm`, and `.frx` files. The manifest is the source of truth for which CommonModules entries are installed in each document. `common-module list`, `common-module update`, and publish-time CommonModules classification use the manifest instead of inferring installation from source file presence. `build`, `publish`, and `doctor` match installed CommonModules source files by exported file name anywhere in the **DocumentSourceSet** because source identity is flat even when source files are organized under nested directories.

`common-module add` updates the manifest and copies the selected CommonModules source files into the target **DocumentSourceSet** in one command. It resolves CommonModules dependencies from the `xls-common-devtools`-owned machine-readable manifest and installs required dependencies together with the explicitly requested entries. The manifest records whether each entry was explicitly requested with `requested: true` or installed as an automatically resolved dependency with `requested: false`, so future auto-remove or prune behavior can avoid treating dependency removal semantics as a breaking manifest change. Adding an already-installed entry succeeds without adding duplicate manifest entries or copying files again. If an already-installed dependency entry is later named explicitly in `common-module add`, the manifest upgrades that entry from `requested: false` to `requested: true` without recopying files. When the entry is not installed but a target source file with the same exported file name exists anywhere in the **DocumentSourceSet**, `common-module add` fails by default to avoid overwriting project-owned source. `--force` permits overwriting during add when exactly one matching exported file name exists; when no matching file exists, add copies to the **CommonModulesDirectory** using the entry's extension-including file name; when multiple matching file names exist, add fails because the flat source identity is ambiguous. `common-module update` overwrites source files for manifest-listed **InstalledCommonModule** entries in place when exactly one matching exported file name exists, copies missing entries to the **CommonModulesDirectory** using the entry's extension-including file name, and fails when multiple matching file names exist. Copy and update output reports the actual `DocumentSourceSet`-relative destination path, including `common-modules/` for new placements and any existing nested path for in-place overwrites. For invocations that affect multiple entries, including dependency-expanded `common-module add`, the command computes the complete manifest and file plan for every affected entry before it mutates the manifest, deletes sidecars, or copies files; any preflight failure aborts without partial manifest or file changes. Manifest writes happen after all planned file deletes and copies succeed, so file operation failures do not leave a manifest that points at missing or partially written CommonModules source. If file delete or copy fails after file mutation begins, the command reports that the manifest was not saved and source files may have been partially updated, and it does not attempt automatic file rollback. If manifest saving fails after successful file operations, the command saves the planned manifest content as a UTF-16LE-with-BOM `vba-project.failed-YYYYMMDD-HHMMSS-fff.json` recovery file beside `vba-project.json`, prints only that recovery file path for manual recovery, and does not attempt automatic file rollback. If saving the recovery file also fails, the command reports that the recovery file could not be saved and does not print the planned manifest JSON to the console. When a CommonModules entry is a `.frm` form, `common-module add` and `common-module update` first remove every existing same-name `.frx` under the target **DocumentSourceSet**. If the CommonModules repository contains the matching `.frx` sidecar, they then copy the `.frm` and `.frx` as one source unit into the same destination directory selected for the `.frm`; if the repository has no matching `.frx`, the target source set is left without a same-name sidecar. Duplicate `.frx` files alone do not define exported source identity.

**VbaProjectReferences** belong to the document definition in `vba-project.json`, under `documents.<document-name>.references`, as objects with a `name` field. `reference add` edits the manifest only; workbook-backed commands such as `build` and `publish` apply the manifest-defined references when they create generated workbooks. Reference names are stored as the human-visible `Reference.Description`-style library name. User input is trimmed and matched case-insensitively, but the manifest stores the reference name as a plain description string. Reference names are unique within one document definition, and adding an already-listed reference succeeds without adding a duplicate. The manifest stores the desired reference set only; it does not mark references as protected.

`build`, `publish`, `test`, and `export` use a dedicated hidden Excel COM instance instead of attaching to a user's existing Excel session. `build` and `publish` create a temporary workbook from the source template, flush existing standard modules/classes/forms, normalize **VbaProjectReferences** to the manifest-defined set, import source deterministically, save to a temporary output, and replace the target output only after success. They recursively discover `.bas`, `.cls`, and `.frm` files anywhere under the **DocumentSourceSet**, treat the result as a flat source file set ordered by extension-including exported file name, fail when exported source file names collide case-insensitively, and do not implicitly exclude hidden or tool-named directories. `build` and `publish` do not validate `.frx` placement as a separate precheck; any same-directory `.frx` handling is left to the underlying form import behavior. Reference normalization removes references that are not listed in the manifest and adds references that are listed but missing, while leaving **ProtectedVbaProjectReferences** in place when Office or VBIDE does not allow removal. If an unlisted protected reference remains in the generated workbook, `build`, `publish`, and `doctor` report a warning identifying the document and reference name. Missing or ambiguous **VbaProjectReferences** fail the command before source import, with an error that identifies the document and reference name. `publish` excludes CommonModules entries classified as test-only by the CommonModules manifest and excludes project-local source marked with `'#ExcludePublish`.

`import` performs an **ExplicitWorkbookImport** from a caller-provided source directory into a caller-provided workbook path without resolving project context. It updates the target workbook in place, flushes existing standard modules/classes/forms, leaves document modules in place, recursively imports `.bas`, `.cls`, and `.frm` files from the source directory as a flat source file set ordered by extension-including exported file name, and saves the workbook only after flush and import complete successfully. Relative paths are not used as ordering tie-breakers because duplicate exported source file names fail before the workbook is opened. `import` does not independently enumerate or validate `.frx` files; any same-directory sidecar consumption remains part of the underlying form import behavior. If the source directory contains no `.bas`, `.cls`, or `.frm` files, or contains duplicate exported source file names case-insensitively, `import` fails before opening the workbook. Unlike `build`, `import` does not create a generated output workbook from a source template, does not add, remove, or normalize **VbaProjectReferences** from a manifest, does not resolve CommonModules dependencies, does not interpret `'#ExcludePublish`, and does not validate whether the resulting workbook compiles.

`test` follows `dotnet test` by building before it runs tests by default. `--no-build` skips that build and uses the existing bin workbook. It emits `text` or `ndjson`, with `text` as the default for command-line use. VS Code and other machine consumers should request `--format ndjson` explicitly for stable streaming. `test --no-build` does not add or update **VbaProjectReferences** in an existing bin workbook; the default build step applies manifest-defined references before tests run. `publish` does not require tests to pass; callers can compose `build`, `test`, and `publish` explicitly.

`export` has a project-aware mode and an explicit-workbook mode. When `--from` is omitted, `export` reads the manifest-resolved bin workbook and writes to the manifest-resolved document source set unless `--to` points at another directory. This project-aware mode requires project and document context. When `--from` is provided, `export` performs an **ExplicitWorkbookExport** from the caller-provided workbook path without resolving project context; if `--to` is omitted in this mode, the current working directory is the destination. `--project` and `--document` are rejected with `--from` because project context does not affect an **ExplicitWorkbookExport**. Export cleans the destination when the destination is manifest-owned or when `--to` is specified. When neither condition applies, export writes into the default destination without deleting unrelated existing files, but exported file paths that collide with existing files are overwritten. Cleanup-enabled exports record existing exported source relative paths, export the workbook to a temporary directory first, and only after that export succeeds recursively delete existing `.bas`, `.cls`, `.frm`, and related `.frx` files from the destination. Cleanup deletes files only and does not remove empty directories. If the workbook export fails, the destination is left untouched. Cleanup-enabled exports restore matching exported file names to their previous relative paths after export and write newly exported file names to the destination root. Stale, displaced, or duplicate `.frx` files in a cleanup-enabled destination do not fail export; they are deleted during cleanup, and exported forms write at most one matching `.frx` beside each `.frm`. Non-cleanup exports still write each exported `.frx` beside its `.frm`, but they do not delete displaced `.frx` files elsewhere in the destination. For exported forms, the `.frm` relative path is the form source unit location; the matching `.frx` is restored beside the `.frm` rather than as an independently placed file.

## Consequences

The command model favors predictable project automation over ad hoc workbook manipulation. It also commits the project manifest encoding, source layout, and manifest-resolved output paths early, which reduces ambiguity for future tools but requires explicit encoding handling in the C# implementation.

Excel and VBIDE automation remain a hard runtime dependency for workbook-backed commands. Environment problems should be surfaced through `doctor` and through clear command failures rather than hidden recovery behavior.
