# Workbook-backed project command model

`vba-dev` will model Office macro automation as a Windows-only, workbook-backed command-line workflow centered on a `project.json` project manifest. The manifest identifies the project and primary document, supplies command defaults, and anchors path resolution so commands can behave consistently from project subdirectories and future multi-document projects.

## Decision

Project manifests use `schemaVersion: 1`, are generated as UTF-16LE with BOM, and are the source of truth for resolving the project root, primary document, document source set, bin output, publish output, and default command options. The reader accepts UTF-16LE with BOM and UTF-8 inputs for compatibility, but generated or rewritten manifests normalize to UTF-16LE with BOM.

Project creation follows the `dotnet new <TEMPLATE>` shape. The initial template is `excel`, so users create a project with `new excel`; future Office document kinds can become additional template names. `--output` selects the project root directory. `--name` selects the generated project and document base name, and when omitted it is derived from the output directory name or current directory name.

The initial source layout is document-oriented even though only one primary Excel document is supported at first: `src/<document-name>/` contains the source template `.xlsm` and exported VBA source, while `bin/<document-name>/` and `publish/<document-name>/` contain generated workbooks. Document-level commands accept `--project` and `--document`; omitted values resolve by walking upward to the nearest `project.json` and then using the manifest's `primaryDocument`.

When `new excel` can copy required initial CommonModules from the canonical repository, it records those entries under the generated document's `commonModules` manifest list as well as copying their source files. Required initial baseline entries are recorded with `requested: true`; dependencies pulled in by that baseline are recorded with `requested: false`. Initial projects therefore start with a manifest-backed installed CommonModules set rather than loose copied files.

Common option aliases follow .NET CLI conventions where they do not create ambiguous command surfaces. `--name` has `-n`, `--output` has `-o`, `--document` has `-d`, and `--format` has `-f`. `--project` has no short alias because it is a context selector rather than an output or resource name. `--from` and `--to` also have no short aliases because `-f` belongs to `--format` and `--to` is intentionally distinct from `--output`.

Common modules are copied from a canonical `common_modules_repo`. `new` creates a usable project even when the repository is missing, but reports a warning; `doctor` reports the missing repository as an environment or project problem. `doctor` also checks manifest-defined **VbaProjectReferences** for every document without changing workbooks. For **InstalledCommonModule** entries, `doctor` verifies that names exist in the CommonModules manifest, that dependencies required by `requested: true` roots are present, and that copied source files are consistent with the manifest. Missing or unknown required entries are failures. Unreachable `requested: false` entries and source file drift start as warnings. CommonModules dependencies and classifications are expected to come from a machine-readable manifest owned by `xls-common-devtools`, not from hard-coded `vba-dev` rules or scraped product documentation.

Resource-specific operations follow .NET CLI-style noun-first command groups where possible instead of using one top-level verb that infers a resource from positional names. CommonModules are added with `common-module add`, listed with `common-module list`, and updated with `common-module update`. `common-module add` accepts one or more CommonModules names in a single invocation and is scoped to one document source set. `common-module list` is also document-scoped, defaults to the primary document when `--document` is omitted, identifies the selected document in its output, defaults to human-readable text, and supports `--format json` for automation. `common-module update` updates already-installed CommonModules entries as a project-wide operation; the initial implementation does not need per-module update selection. `common-module remove` is not part of the initial command surface because dependency ownership, shared dependencies, and locally edited copied files need separate deletion semantics. **VbaProjectReferences** use `reference add`, `reference list`, and `reference remove`. These reference commands are scoped to one document definition. `reference add` and `reference remove` accept one or more reference names in a single invocation. `reference list` identifies the selected document in its output, defaults to human-readable text, and supports `--format json` for automation. Because the command surface is not released yet, earlier top-level forms such as `add <module>`, `add module`, `add reference`, `update`, and `remove reference` do not need compatibility support. `reference remove` removes **VbaProjectReferences** from the manifest only and succeeds for references that are already absent.

**InstalledCommonModule** entries belong to the document definition in `project.json`, under `documents.<document-name>.commonModules`, as objects with a `name` field. CommonModules names are extensionless module base names; the CommonModules manifest resolves those names to actual `.bas`, `.cls`, `.frm`, and `.frx` files. The manifest is the source of truth for which CommonModules entries are installed in each document. `common-module list`, `common-module update`, and publish-time CommonModules classification use the manifest instead of inferring installation from source file presence.

`common-module add` updates the manifest and copies the selected CommonModules source files into the target **DocumentSourceSet** in one command. It resolves CommonModules dependencies from the `xls-common-devtools`-owned machine-readable manifest and installs required dependencies together with the explicitly requested entries. The manifest records whether each entry was explicitly requested with `requested: true` or installed as an automatically resolved dependency with `requested: false`, so future auto-remove or prune behavior can avoid treating dependency removal semantics as a breaking manifest change. Adding an already-installed entry succeeds without adding duplicate manifest entries or copying files again. If an already-installed dependency entry is later named explicitly in `common-module add`, the manifest upgrades that entry from `requested: false` to `requested: true` without recopying files. When the entry is not installed but a target source file already exists, `common-module add` fails by default to avoid overwriting project-owned source. `--force` permits overwriting during add. `common-module update` overwrites source files for manifest-listed **InstalledCommonModule** entries because its purpose is to replace installed copies with the canonical repository version.

**VbaProjectReferences** belong to the document definition in `project.json`, under `documents.<document-name>.references`, as objects with a `name` field. `reference add` edits the manifest only; workbook-backed commands such as `build` and `publish` apply the manifest-defined references when they create generated workbooks. Reference names are stored as the human-visible `Reference.Description`-style library name. User input is trimmed and matched case-insensitively, but the manifest stores the reference name as a plain description string. Reference names are unique within one document definition, and adding an already-listed reference succeeds without adding a duplicate. The manifest stores the desired reference set only; it does not mark references as protected.

`build`, `publish`, `test`, and `export` use a dedicated hidden Excel COM instance instead of attaching to a user's existing Excel session. `build` and `publish` create a temporary workbook from the source template, flush existing standard modules/classes/forms, normalize **VbaProjectReferences** to the manifest-defined set, import source deterministically, save to a temporary output, and replace the target output only after success. Reference normalization removes references that are not listed in the manifest and adds references that are listed but missing, while leaving **ProtectedVbaProjectReferences** in place when Office or VBIDE does not allow removal. If an unlisted protected reference remains in the generated workbook, `build`, `publish`, and `doctor` report a warning identifying the document and reference name. Missing or ambiguous **VbaProjectReferences** fail the command before source import, with an error that identifies the document and reference name. `publish` excludes CommonModules entries classified as test-only by the CommonModules manifest and excludes project-local source marked with `'#ExcludePublish`.

`test` follows `dotnet test` by building before it runs tests by default. `--no-build` skips that build and uses the existing bin workbook. It emits `text` or `ndjson`, with `text` as the default for command-line use. VS Code and other machine consumers should request `--format ndjson` explicitly for stable streaming. `test --no-build` does not add or update **VbaProjectReferences** in an existing bin workbook; the default build step applies manifest-defined references before tests run. `publish` does not require tests to pass; callers can compose `build`, `test`, and `publish` explicitly.

`export` defaults to reading the manifest-resolved bin workbook and writing to the manifest-resolved document source set. `--from` may point at another workbook, and `--to` may point at another directory. The default document source set export deletes existing `.bas`, `.cls`, `.frm`, and related `.frx` files before export; explicit `--to` exports do not clean the destination.

## Consequences

The command model favors predictable project automation over ad hoc workbook manipulation. It also commits the project manifest encoding and source layout early, which reduces ambiguity for future tools but requires explicit encoding handling in the C# implementation.

Excel and VBIDE automation remain a hard runtime dependency for workbook-backed commands. Environment problems should be surfaced through `doctor` and through clear command failures rather than hidden recovery behavior.
